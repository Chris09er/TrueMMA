Die sechs Wege

Jeder Weg ist eine Kette. Wenn etwas nicht funktioniert, ist fast immer genau eine dieser Ketten unterbrochen — die Frage lautet dann nur, an welchem Glied.

Daten rein Daten raus Benachrichtigung Ausliefern Auth-Mail

1 · Daten rein — der Sync

Läuft alle 6 Stunden (prod). Holt Events, Fighter und Fights und schreibt sie idempotent per external_id in Supabase.

balldontlie/leagues, /events, /fights
GitHub Actionssync-balldontlie-full
SupabaseService-Role-Key, umgeht RLS
Der Live-Sync ist derselbe Weg, nur schmaler. Er prüft erst per DB-Abfrage, ob überhaupt ein Event läuft, und holt nur dann dessen Fights nach — sonst beendet er sich sofort, ohne balldontlie anzufassen.

2 · Daten raus — die App liest

Kein Backend dazwischen. Die App spricht PostgREST direkt an; was sie sehen darf, entscheidet allein RLS.

Alle Inhalts-Tabellen sind für die App nur lesbar. Genau drei Ausnahmen schreiben anonym: Fighter-Follows, Liga-Follows und Fight-Votes — jeweils an einer selbst erzeugten Geräte-Kennung verankert, nicht an einem Login.

Warum zweiphasig? pg_net arbeitet asynchron und startet den Request erst nach dem Commit — ein Durchlauf kann also gar nicht wissen, ob der Push ankam. Deshalb gleicht jeder Tick zuerst das Ergebnis des vorherigen ab, bevor er neue Requests stellt. Erst eine bestätigte 2xx-Antwort markiert das Event als erledigt.

Beide oberen Pfade versenden seit 2026-07-19 in ≤100er-Blöcken (Expos Limit pro Request) über eine gemeinsame Funktion, die auch die dritte Zeile hier speist: Expo liefert HTTP 200 selbst wenn einzelne Nachrichten fehlschlagen, daher fragt ein dritter, unabhängiger pg_cron-Job Zustellquittungen ab und räumt deinstallierte Geräte auf.

Der Event-Reminder ist bewusst kein Push: Er wird lokal auf dem Gerät geplant und braucht überhaupt keinen Server.

4 · Ausliefern — von Branch zu Gerät

Ein Push auf stage oder main löst zwei unabhängige Wege aus, je nachdem was sich geändert hat.

git pushstage / main
deploy-migrationsbei supabase/migrations/**
SupabaseSchema wandert mit

Ein Build zieht nur Updates seines eigenen Channels. Passen App-Version und Update nicht zusammen, wird nichts ausgeliefert — das ist die Bremse, die verhindert, dass ein JS-Update auf einem Binary landet, dessen nativer Teil nicht dazu passt.

5 · Auth-Mail — Registrierung und Passwort-Reset

Der kürzeste Weg, aber der mit der unerwartetsten Begründung.

Supabase AuthSignup / Reset
IONOS SMTPsmtp.ionos.de:465
Postfach6-stelliger Code
Der Passwort-Reset schickt bewusst einen Code statt eines Links. Ein Link bräuchte Deep Linking, also ein URL-Schema in der nativen Config — und damit einen neuen Build. Der Code umgeht das komplett. Die zugehörige Mail-Vorlage lebt nur im Supabase-Dashboard, in keinem Repo.

6 · Warteliste — der einzige Weg ohne App dazwischen

Eine separate, statische Seite außerhalb der Expo-App schreibt Anmeldungen direkt in Supabase.

Kosten & Limits

Laufende Kosten: 9,99 $ im Monat, ausschließlich für balldontlie. Alles andere ist Free Tier — und die Grenzen sind nach den Optimierungen weit entfernt.

Warum das Repo öffentlich ist: Nur so sind die Runner-Minuten unbegrenzt frei. Bei einem privaten Repo würde allein der häufige Live-Check die 2.000 Freiminuten sprengen — GitHub rechnet pro Lauf mindestens eine volle Minute ab, auch wenn das Skript sofort abbricht.

Bekannte Lücken

Nichts davon blockiert die Entwicklung — aber alles davon sollte vor einem Store-Release entschieden sein.

bewusst

Votes und anonyme Follows sind manipulierbar

Die Schreibrechte für anonyme Zeilen sind absichtlich weit: ohne serverseitig prüfbare Geräte-Identität gibt es keine saubere Alternative. Abstimmungsergebnisse sind daher indikativ, nicht belastbar. Anonyme Push-Tokens sind über den anon-Key auslesbar.

offen

League-Start-Push nie echt zugestellt

Die Mechanik ist lokal und auf stage verifiziert, der Expo-Aufruf selbst wurde aber nur simuliert. Auf stage folgt derzeit niemand einer Liga — ohne das lässt sich der Weg nicht real auslösen.

offen

Datenschutzerklärung nicht veröffentlicht

Entwurf liegt zweisprachig vor, braucht aber juristische Prüfung. Beide Stores verlangen bei Einreichung eine öffentlich erreichbare URL — die Seite ist vorbereitet, aber bewusst nirgends deployt.

später

Supabase Branching statt zwei Projekten

Zwei Free-Tier-Projekte kosten 0 € statt rund 35 €/Monat. Sobald prod ohnehin Pro-Features braucht — etwa bessere Backups vor einem Release — wird der Wechsel attraktiv und ist dann billig, weil stage nur Wegwerfdaten hält.

offen

Landingpage geschrieben, aber nirgends gehostet

website/index.html existiert und schreibt schon erfolgreich in waitlist_signups (prod), ist aber noch nicht unter true-mma.com live geschaltet — reines Deployment, kein Code-Problem.
